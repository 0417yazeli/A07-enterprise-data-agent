## 1. 产品概述

### 1.1 一句话定位

> **面向制造业业务人员的"自然语言问数 + 智能分析"平台，让不懂 SQL 的人也能在 30 秒内完成一次业务分析。**

### 1.2 产品愿景

将企业数据底座从"工程师的工具箱"变成"业务人员的对话伙伴"。业务人员用自然语言提问，系统自动完成数据定位、SQL 生成、查询执行、图表绘制、结论生成的全流程闭环。

### 1.3 核心价值主张

| 维度 | 现状 | 目标 |
|------|------|------|
| 分析响应时间 | 3-5 天（排期等数据工程师） | **30 秒**（自助即时） |
| 业务人员自助率 | 不到 20% | **80%+** |
| 数据工程师重复劳动 | 40% 时间在取数 | **大幅释放** |
| 分析门槛 | 必须懂 SQL/Python | **自然语言即可** |
| 结果呈现 | 原始数据表格 | **表格 + 图表 + 文字结论 + 报告** |

### 1.4 核心闭环

```
用户提问（自然语言）
     ↓
智能理解（LLM 意图识别 + 业务知识注入）
     ↓
自动执行（NL2SQL → 数据查询 → 指标计算 → 可选机器学习）
     ↓
结果展示（表格 + ECharts 图表 + 文字结论 + Markdown 报告）
     ↓
反哺业务（沉淀问句、指标口径、分析模板，持续优化）
```

## 2. 背景与问题分析

### 2.1 行业背景

- 2025 年国家《"十四五"智能制造发展规划》收官，**90% 规上制造业企业已完成 ERP/MES/QMS 数字化建设**
- 杭州自动化技术研究院服务的浙江制造业集群已普遍具备数据底座基础
- 行业共识：**"数据采集容易，分析困难"** — 80% 数据处于沉睡状态
- 大语言模型 2023-2025 年发生质变：从"会聊天"到"会用工具"，NL2SQL 基准突破 70%

### 2.2 核心痛点（5 项）

| 编号 | 痛点 | 影响 | 量化 |
|------|------|------|------|
| P1 | 业务人员难以理解数据表/字段/指标 | 自助分析门槛极高 | **80%+ 业务人员放弃自助** |
| P2 | 提出业务问题后难以快速定位数据资源 | 需求响应慢 | 数据团队人均响应 **3-5 天** |
| P3 | 统计/指标/图表生成依赖人工 | 大量重复劳动 | 分析师 **40%** 时间在取数 |
| P4 | 数据分析依赖 SQL/Python，门槛高 | 人才严重不足 | 制造业数据工程师**缺口 30 万+** |
| P5 | 分析结果需人工整理，缺乏直观展示 | 决策延迟 | 一份周报从数据到发布 **8 小时+** |

**5 项痛点共同指向一个核心矛盾：数据丰富但分析能力稀缺。**

### 2.3 用户原声

> "现场人员不懂 SQL，质量经理想看良率需要排期 3 天等数据工程师响应。"
> —— 某合作制造企业 CIO（2024 行业论坛）

> "当生产经理晚上 9 点想看今天良率时，他不应该需要等到明天数据工程师上班。"
> —— 命题方一线访谈

## 3. 用户画像与使用场景

### 3.1 核心用户角色

| 角色 | 典型需求 | 技术能力 | 使用频率 |
|------|---------|---------|---------|
| **生产主管** | 查看各工序产量趋势、良率变化 | 会用 Excel，不懂 SQL | 每日 |
| **质量工程师** | 分析缺陷类型、不良数、检验结果 | 懂质量业务，不懂 SQL | 每日/每周 |
| **设备工程师** | 分析停机时间、报警次数、运行效率 | 懂设备，不懂编程 | 每周 |
| **工厂管理者** | 快速生成周报/月报/专题分析 | 只看结果，不懂技术 | 每周/每月 |

### 3.2 典型使用场景

| 场景 | 用户角色 | 典型问句 | 期望输出 |
|------|---------|---------|---------|
| **S1 良率分析** | 生产主管 | "请分析各工序的良率" | 表格 + 柱状图 + 良率排序 |
| **S2 不良品排查** | 质量工程师 | "找出最近一个月不良数量最高的 5 个产品" | 表格 + 饼图 + 缺陷分布 |
| **S3 产量趋势** | 生产主管 | "统计每条产线最近 7 天的产量趋势" | 多线折线图 + 环比变化 |
| **S4 异常停机** | 设备工程师 | "帮我找出近期异常停机的设备" | 异常设备 Top5 + 时间分布图 |
| **S5 质量周报** | 管理者 | "生成一份本周质量分析结果" | 表格 + 图表 + 文字结论 + Markdown |
| **S6 不良归因** | 质量工程师 | "分析设备停机时间和不良率是否相关" | 散点图 + 相关系数 + 业务解读 |
| **S7 库存分析** | 管理者 | "查看当前各物料库存周转情况" | 表格 + 柱状图 + 预警高亮 |

### 3.3 用户旅程

```
打开应用 → 选择身份 → 输入自然语言问题
  → 等待系统理解（流式反馈） → 查看表格 + 图表 + 文字结论
    → 进一步追问（切换维度/时间范围） → 收藏问句 → 导出报告 → 分享同事
```

├───────────┬───────────┬───────────┬───────────┬────────────┤
│ ① 业务知识 │ ② 数据资源 │ ③ 自然语言 │ ④ SQL/ML  │ ⑤ 分析结果  │
│   管理     │   理解     │  智能分析   │ 生成执行  │   展示      │
├───────────┴───────────┴───────────┴───────────┴────────────┤
│                    平台基础（Agent/LLM/用户/日志）            │

### 4.2 功能需求 ①：业务知识管理

**需求描述**：系统能够围绕制造业务场景，整理和维护业务对象、业务指标、业务规则和分析主题，并以知识图谱方式可视化展示。

**功能点**：

| 编号 | 功能 | 描述 | 优先级 |
|------|------|------|--------|
| FK-1.1 | 业务对象管理 | 维护业务对象（工单/工序/产品/设备/物料/产线等），含业务标签、口径定义 | P0 |
| FK-1.2 | 业务指标管理 | 维护指标定义（良率/不良率/停机率/库存周转率等），含计算公式、关联表/字段 | P0 |
| FK-1.3 | 业务规则管理 | 维护业务规则（如"良率 = 合格数/投入数 × 100%"），作为 NL2SQL 的校准依据 | P1 |
| FK-1.4 | 分析主题管理 | 维护 4 大分析主题（生产/质量/设备/库存），含主题描述与关联指标 | P0 |
| FK-1.5 | 知识图谱可视化 | ECharts 关系图展示"业务对象 ↔ 数据表 ↔ 字段"关联，hover 显示详情 | P1 |
| FK-1.6 | 自然语言检索 | 用户输入"良率怎么算"，返回口径定义 + SQL 模板 + 关联表 | P1 |
| FK-1.7 | 基础 CRUD | 所有业务知识实体支持创建、读取、更新、删除 | P1 |

**业务知识范围**（v1.0）：

| 分析主题 | 业务对象 | 核心指标 | 涉及表 |
|---------|---------|---------|--------|
| 生产分析 | 工单/工序/产线/产品 | 产量/良率/工时/效率 | workorder / process / route |
| 质量分析 | 检验/缺陷/产品 | 不良率/合格率/缺陷分布 | inspection / defect / standard |
| 设备分析 | 设备/停机/报警 | 停机率/MTBF/MTTR/报警频率 | equipment / downtime / alarm |
| 库存分析 | 物料/仓库/交易 | 库存周转率/库存天数/缺料率 | material / balance / transaction |

### 4.4 功能需求 ③：自然语言智能分析（核心）

**需求描述**：用户通过自然语言输入分析需求，系统理解意图并自动完成数据查询、统计分析和指标计算。这是整个系统的核心能力。

**功能点**：

| 编号 | 功能 | 描述 | 优先级 |
|------|------|------|--------|
| FK-3.1 | 自然语言输入 | 聊天式输入框，支持自由文本输入 | P0 |
| FK-3.2 | 意图识别 | LLM 识别用户意图（查询/统计/预测/异常/报告） | P0 |
| FK-3.3 | NL2SQL | 自动生成 SQL 语句，支持多表 JOIN、GROUP BY、聚合函数、子查询 | P0 |
| FK-3.4 | Schema Linking | 自动将业务术语映射到具体的表/字段（"良率"→ quality_inspection.pass_rate） | P0 |
| FK-3.5 | 业务口径注入 | 将"良率 = 合格数/投入数"等口径作为 LLM system prompt 校准生成结果 | P1 |
| FK-3.6 | SQL 安全执行 | 只允许 SELECT/WITH，强制 LIMIT 1000，白名单校验 | P0 |
| FK-3.7 | 流式输出 | SSE 流式返回分析进度（"正在理解..."→"查询中..."→"生成图表..."） | P1 |
| FK-3.8 | 多轮对话 | 支持追问（"那近 7 天的趋势呢？"→ 自动切换时间范围 + 图表类型） | P2 |
| FK-3.9 | 错误处理 | SQL 生成失败时自动反思重试（最多 3 次），失败则引导用户换个问法 | P1 |
| FK-3.10 | 问句模板库 | 预置 20 条常用自然语言问句，一键点击发送 | P1 |
| FK-3.11 | 代码可见 | 用户可查看系统生成的 SQL（点击"查看 SQL"展开） | P2 |

**v1.0 支持的分析类型**：

| 分析类型 | 示例问句 | 预期 SQL 模式 |
|---------|---------|-------------|
| 简单聚合 | "各工序的良率" | `SELECT process, AVG(pass_rate) ... GROUP BY process` |
| 条件筛选 | "不良数最高的 5 个产品" | `SELECT ... WHERE ... ORDER BY ... LIMIT 5` |
| 时间趋势 | "近 7 天产量趋势" | `SELECT date, SUM(qty) ... WHERE date >= ... GROUP BY date` |
| 多表关联 | "A 产品各工序不良率" | 3 表 JOIN（product + process + defect） |
| 对比分析 | "本周 vs 上周良率" | 子查询 / CTE 实现周环比 |
| 排名 TopN | "停机时间最长的 10 台设备" | `ORDER BY ... DESC LIMIT 10` |

## 5. 非功能需求

### 5.1 性能要求

| 指标 | 目标值 | 备注 |
|------|--------|------|
| API 响应时间（P50） | < 3 秒（含 LLM 调用） | 简单查询 |
| API 响应时间（P95） | < 10 秒 | 复杂多表 JOIN + 解读 |
| SQL 执行超时 | 30 秒 | 超时自动终止 |
| 并发用户 | ≥ 10（演示场景） | 单机部署 |
| LLM 流式首字节 | < 2 秒 | SSE 流式输出 |

### 5.2 安全要求

| 要求 | 实现 |
|------|------|
| SQL 注入防护 | 只允许 SELECT / WITH / EXPLAIN，禁止 DDL/DML/DCL |
| API Key 保护 | 走环境变量（`.env`），不入库、不入 Git |
| 数据脱敏 | 不展示敏感 PII 字段 |

### 6.2 Agent 工作流（ReAct 范式）

```
用户问题
  ↓
Thought（理解意图：用户想查询什么？涉及哪些表？）
  ↓
Action（选择并调用工具：list_tables → get_schema → execute_sql）
  ↓
Observation（观察工具返回结果：数据是否正确？是否完整？）
  ↓
  ├── 未完成 → 回到 Thought（需要更多工具？SQL 需要修正？）
  ↓
  └── 已完成 → Final Answer（生成业务解读 + 结构化输出）
                  ↓
              输出：表格 + 图表配置 + 文字结论
```

### 6.3 核心数据流

```
自然语言问句 + 业务知识 + Schema 元数据
  → Prompt 拼接
    → LLM 推理
      → 生成 SQL
        → SQL 安全校验（白名单 + LIMIT）
          → SQL 执行（PostgreSQL）
            → 数据结果
              → LLM 业务解读
                → 结构化输出
                  → 表格 + 图表 + 文字结论 + Markdown 报告
```

### 8.3 核心业务指标口径

| 指标 | 口径 | 涉及表 |
|------|------|--------|
| **良率** | pass_qty / (pass_qty + scrap_qty) × 100% | process |
| **不良率** | fail_qty / inspect_qty × 100% | inspection |
| **设备停机率** | SUM(duration_min) / 总运行时间 × 100% | downtime |
| **MTBF** | 总运行时间 / 故障次数 | downtime |
| **MTTR** | SUM(维修时间) / 维修次数 | maintenance |
| **库存周转率** | 出库总量 / 平均库存 | transaction / balance |
| **工序达成率** | actual_qty / plan_qty × 100% | process / workorder |

|------|------|------|---------|

| **演示超时** | 中 | 中 | 主版 10 分钟 + 精简版 5 分钟备选 |
| **代码冲突** | 中 | 中 | 分支策略 + 每天合并 + 路径占位符规范 |
| **命题理解偏差** | 中 | 中 | M1 周日复盘验证方向 + 与命题方沟通 |
